3. Google Ads API Usage

3.1 Purpose

SolarZap uses the Google Ads API exclusively for Offline Conversion Tracking. When a lead that originated from a Google Ads click progresses through the sales pipeline (e.g., schedules a visit, closes a sale), SolarZap automatically uploads these offline conversion events back to Google Ads. This enables our users to:

  • Measure the true ROI of their Google Ads campaigns
  • Optimize bidding strategies based on actual sales outcomes
  • Identify which keywords and ads drive real revenue

3.2 API Features Used

API Feature Purpose Frequency
ListAccessibleCustomers List Google Ads accounts accessible to the authenticated user during initial setup On-demand (setup only)
CustomerService (GAQL) Query customer hierarchy to map MCC → child accounts On-demand (setup only)
ConversionActionService (GAQL) List available conversion actions for the selected customer account On-demand (setup only)
ConversionUploadService.UploadClickConversions Upload offline conversion events (pipeline stage changes mapped to conversions) Event-driven, typically 5-50 per org per day

3.3 Data Flow

1. User clicks Google Ad → lands on client's website 2. SolarZap tracking script captures gclid/gbraid/wbraid 3. Lead is created in SolarZap CRM with click ID attached 4. Lead progresses through pipeline (e.g., "Qualified" → "Visit Scheduled" → "Proposal Sent" → "Sale Closed") 5. Each configured stage triggers a conversion event 6. SolarZap conversion-dispatcher sends the event to Google Ads API via UploadClickConversions 7. Google Ads receives the offline conversion and attributes it to the original click

4. Authentication & Authorization

4.1 OAuth 2.0 Flow

SolarZap implements the standard OAuth 2.0 Authorization Code Flow with offline access:

User clicks "Connect Google Ads" │ ▼ SolarZap Backend generates OAuth URL (scope: adwords, access_type: offline, prompt: consent) │ ▼ User is redirected to Google's consent screen │ ▼ User authorizes SolarZap │ ▼ Google redirects to SolarZap callback with authorization code │ ▼ SolarZap exchanges code for access_token + refresh_token │ ▼ refresh_token is stored in Supabase Vault (encrypted) access_token is used immediately, then discarded │ ▼ User selects MCC → Customer Account → Conversion Action │ ▼ Configuration is saved to the database

4.2 Token Management

Token Type Storage Lifecycle
Access Token In-memory only (server-side) Refreshed on each API call, never persisted
Refresh Token Supabase Vault (encrypted at rest) Stored per-organization, rotated on re-authorization
Developer Token Supabase Edge Function secret Global, managed by Arkan Labs
Client ID/Secret Supabase Edge Function secrets Global, managed by Arkan Labs

4.3 Token Security

  • All tokens are handled exclusively on the server-side (Supabase Edge Functions / Deno runtime)
  • Refresh tokens are never exposed to the client/browser
  • Refresh tokens are stored in Supabase Vault with encryption at rest
  • Old refresh tokens are automatically deleted when a user re-authorizes
  • Token revocation is attempted on disconnect

6. User Experience

6.1 Setup Flow

  1. User navigates to Settings → Tracking & Conversions → Platforms
  2. User clicks "Connect Google Ads" button
  3. User is redirected to Google OAuth consent screen
  4. User authorizes SolarZap to access their Google Ads account
  5. User is redirected back to SolarZap with a success notification
  6. Dropdowns populate with the user's accessible accounts
  7. User selects: MCC Account → Customer Account → Conversion Action
  8. User clicks "Save Selection" and optionally "Test Connection"
  9. Integration is active — conversions are uploaded automatically

6.2 Ongoing Usage

Once configured, the integration runs fully automatically. As leads progress through the CRM pipeline:

  • Stage changes trigger conversion events
  • The conversion-dispatcher retrieves credentials from Vault
  • Offline conversions are uploaded to Google Ads via the API
  • Failed uploads are retried with exponential backoff (30s → 60s → 5m → 30m → 1h)
  • Users can monitor delivery status on the "Entregas" tab

7. Rate Limits & Compliance

7.1 Expected API Usage

Metric Expected Value
Organizations using Google Ads 10-100 (growing)
Conversion uploads per org per day 5-50
Total API calls per day ~500-5,000
Setup API calls (list accounts) Occasional (on-demand)

7.2 Compliance

  • SolarZap does NOT modify campaigns, ad groups, ads, or bidding strategies
  • SolarZap does NOT access user data beyond what is necessary for conversion tracking
  • SolarZap does NOT store Google Ads data locally (beyond the configuration IDs)
  • All API interactions are server-side only (no client-side API calls)
  • Privacy Policy: https://solarzap.com.br/privacidade
  • Terms of Service: https://solarzap.com.br/termos
  • Compliant with LGPD (Brazil's General Data Protection Law)

8. Error Handling & Monitoring

  • Token expiration: Access tokens are refreshed on every API call using the stored refresh token
  • Failed uploads: Automatic retry with exponential backoff (up to 5 attempts)
  • Invalid credentials: Users are prompted to re-authorize via the "Reconnect" flow
  • Rate limiting: Respected via retry-after headers
  • Monitoring: All conversion deliveries are logged with status tracking (pending → processing → sent/failed)
